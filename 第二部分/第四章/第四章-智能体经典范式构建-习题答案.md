## 1. 三种智能体范式的区别与组合

### 题目整理

本题要求分析 `ReAct`、`Plan-and-Solve` 和 `Reflection` 在“思考”和“行动”组织方式上的本质区别；以智能家居控制助手为例选择合适范式；并设计三种范式的混合架构与适用场景。

### 参考答案

三种范式的核心差异可以从“何时思考、何时行动、如何修正”理解：

| 范式 | 思考方式 | 行动方式 | 适合任务 |
| --- | --- | --- | --- |
| `ReAct` | 每一步先思考，再决定是否调用工具 | 思考、行动、观察交替进行 | 需要外部工具、状态不确定、可边做边查的任务 |
| `Plan-and-Solve` | 先生成完整计划，再逐步执行 | 执行阶段通常按计划推进 | 目标清晰、步骤可拆解、逻辑链较稳定的任务 |
| `Reflection` | 先生成结果，再评审和优化 | 行动后进入反思和修订循环 | 对质量、鲁棒性、代码或文本质量要求高的任务 |

如果设计“智能家居控制助手”，我会选择混合架构，而不是单一范式。基础控制层适合 `ReAct`，因为它需要读取传感器状态、调用设备控制 API，并根据设备反馈继续调整。例如用户说“有点热”，系统需要查询室温、空调状态、用户偏好，再决定降温还是开窗帘。

混合架构可以这样设计：

1. `Plan-and-Solve` 负责高层计划：把“睡前模式”拆成关灯、调空调、拉窗帘、设置安防等步骤。
2. `ReAct` 负责每个步骤的工具调用：查询设备状态、调用控制接口、读取执行结果。
3. `Reflection` 负责事后优化：如果用户多次手动改回温度，系统反思当前策略不符合偏好，更新用户习惯。

这种组合适合家庭自动化、客服工单处理、旅行规划、企业流程自动化等场景：任务既有计划性，又需要工具交互，还需要从失败和反馈中持续改进。

## 2. ReAct 输出解析的脆弱性与改进

### 题目整理

本题要求分析 4.2 节中用正则表达式解析 `Thought`、`Action` 等输出的潜在问题，提出更鲁棒的解析方案，并尝试改造代码，对比两种方案的优缺点。

### 参考答案

正则解析的主要问题是对格式高度敏感。模型只要多输出一段解释、少写一个冒号、把 `Action Input` 改成 `ActionInput`，或者在自然语言中也出现类似字段名，解析就可能失败。更麻烦的是，工具参数如果包含换行、括号、JSON、Markdown 代码块，简单正则很容易截断或误匹配。

更可靠的方案包括：

- 使用 JSON 输出，并对字段做 schema 校验。
- 使用函数调用或工具调用协议，由模型直接返回结构化调用。
- 使用 XML/YAML 等显式标签格式，但仍需解析和校验。
- 使用有限状态机解析固定协议，并在解析失败时要求模型按格式重试。

推荐在工程中优先使用 JSON 或工具调用协议。例如要求模型输出：

```json
{
  "thought": "需要查询天气。",
  "action": {
    "name": "search",
    "input": "杭州明天天气"
  },
  "final_answer": null
}
```

这种方式的优点是字段边界清晰，便于校验、重试、记录和监控；缺点是提示词要更严格，部分模型可能输出非法 JSON，需要加入修复或重试机制。

示例实现见 [react_structured_parser.py](D:\code_local\hello-agents-notes\第二部分\第四章\code\react_structured_parser.py)。

## 3. 为 ReAct 添加计算器工具与工具失败处理

### 题目整理

本题要求扩展 4.2.2 节中的 `ToolExecutor`：为 `ReAct` 智能体添加计算器工具；设计工具选择失败或参数错误时的纠正机制；并思考工具数量增长到几十甚至上百个时，如何优化工具组织和检索。

### 参考答案

计算器工具应该和搜索工具分工明确。搜索工具用于获取外部事实，计算器工具用于确定性数学计算。对于表达式 `(123 + 456) * 789 / 12`，不应让模型心算，而应调用安全的表达式求值器。

工具失败处理可以分三层：

1. 参数校验：工具执行前检查表达式是否合法、是否包含危险语法。
2. 错误反馈：将失败原因作为 `Observation` 返回给模型，例如“参数不是合法数学表达式，请只传入表达式字符串”。
3. 失败计数：连续失败达到阈值后，强制模型重新选择工具，或触发人工兜底。

当工具数量达到 50 到 100 个时，简单把所有工具描述塞进提示词会带来上下文浪费、选择混乱和成本上升。工程上可以做工具分层：

- 按领域分组：订单、支付、物流、售后、内容、安全等。
- 先检索再注入：用关键词或向量检索选出最相关的 3 到 8 个工具。
- 建立工具路由器：先让一个轻量模型判断任务类别，再暴露对应工具集。
- 为工具维护 schema、示例、权限级别、幂等性和风险等级。

示例实现见 [calculator_tool.py](D:\code_local\hello-agents-notes\第二部分\第四章\code\calculator_tool.py)。

## 4. Plan-and-Solve 的动态重规划与分层规划

### 题目整理

本题要求分析静态计划在执行失败时的不足，设计动态重规划机制；比较 `Plan-and-Solve` 与 `ReAct` 在商务旅行预订任务中的适配性；并设计高层计划加子计划的分层规划系统。

### 参考答案

静态计划的问题是执行阶段默认计划始终有效，但真实环境会变化。例如预订机票时，原计划中的航班可能售罄；酒店价格可能超预算；租车接口可能返回失败。动态重规划机制应包含：

- 执行状态记录：每一步记录输入、输出、失败原因和环境变化。
- 失败分类：区分参数错误、工具不可用、业务约束冲突、外部资源变化。
- 局部重规划：优先只修改失败步骤及其后续步骤，而不是推翻全部计划。
- 全局重规划：当目标、预算或关键约束变化时，重新生成完整计划。
- 停止条件：限制重规划次数，避免无限循环。

对于“北京到上海商务旅行预订”任务，推荐 `Plan-and-Solve + ReAct`。高层上，它需要规划机票、酒店、租车和日程安排，因此适合先规划；执行中又必须查询实时库存、价格和用户偏好，因此每个子任务适合用 `ReAct` 调工具。

分层规划可以分为：

```text
目标：完成商务旅行预订
高层计划：
1. 确认行程约束
2. 预订交通
3. 预订酒店
4. 安排本地交通
5. 汇总并确认

子计划示例：预订交通
1. 查询可选航班
2. 按时间、价格、舱位筛选
3. 向用户确认
4. 下单并保存票据
```

分层规划的优势是可读性强、便于局部替换，也方便把不同子任务交给不同工具或子智能体处理。

示例实现见 [dynamic_planner.py](D:\code_local\hello-agents-notes\第二部分\第四章\code\dynamic_planner.py)。

## 5. Reflection 机制的模型选择、终止条件与论文写作助手

### 题目整理

本题要求思考 Reflection 中执行模型和反思模型是否应相同；评估“无需改进/达到最大次数”的终止条件；并为学术论文写作助手设计多维度反思机制。

### 参考答案

使用不同模型会带来明显的工程权衡。强模型做反思、快模型做执行，通常可以降低成本并提高评审质量。反思模型负责发现逻辑漏洞、事实风险、结构问题；执行模型负责根据反馈生成修订稿。缺点是不同模型的风格、格式遵循能力和判断标准可能不一致，需要统一输出 schema 和评价尺度。

“反馈包含无需改进”或“达到最大迭代次数”是可用但粗糙的终止条件。更智能的终止机制可以结合：

- 质量评分：反思模型给出结构化分数，例如正确性、完整性、可读性均高于阈值。
- 改进幅度：连续两轮改动很小，则停止。
- 风险等级：若仍存在高风险问题，继续迭代或转人工。
- 成本预算：达到 token、时间或调用次数上限时停止。
- 测试结果：代码任务以单元测试、静态检查或 benchmark 作为终止依据。

论文写作助手的多维度 Reflection 可以设计为多个评审器：

| 评审维度 | 检查内容 | 输出 |
| --- | --- | --- |
| 段落逻辑 | 论点是否清晰，段落之间是否衔接 | 逻辑问题和改写建议 |
| 方法创新性 | 是否说清方法相对已有工作的差异 | 创新点强化建议 |
| 实验可信度 | 指标、数据集、基线是否充分 | 缺失项和补充建议 |
| 语言表达 | 是否学术、简洁、一致 | 句式和术语修改 |
| 引用规范 | 结论是否有引用支撑 | 需要补引用的位置 |
| 学术诚实 | 是否夸大结果或引入无依据结论 | 风险标注 |

整体流程可以是：生成初稿 -> 多维度反思 -> 合并反馈 -> 局部重写 -> 引用校验 -> 最终润色。高风险内容必须要求用户确认或补充证据。

## 6. 提示词结构、角色设定与 few-shot 示例

### 题目整理

本题要求比较 `ReAct` 与 `Plan-and-Solve` 提示词结构差异；修改 `Reflection` 中的角色设定并观察影响；为某个智能体添加 `few-shot` 示例并比较效果。

### 参考答案

`ReAct` 提示词强调循环协议，通常要求模型按“思考、行动、行动输入、观察”的格式输出。它服务于工具交互，重点是让模型在每一步明确是否需要调用工具，以及如何根据观察继续行动。

`Plan-and-Solve` 提示词强调先拆解任务，再执行计划。它通常要求模型输出步骤列表或结构化计划，重点是提升复杂推理任务的条理性，减少边做边想导致的遗漏。

`Reflection` 的角色设定会显著影响反馈方向。例如：

- “极其严格的算法工程师”会更关注复杂度、性能和边界条件。
- “注重可读性的开源维护者”会更关注命名、注释、模块划分和可维护性。
- “安全审计专家”会更关注输入校验、权限、隐私和危险操作。

few-shot 示例可以用于提升格式遵循能力。以 `ReAct` 为例：

```text
示例：
问题：计算 12 * 8 的结果
输出：
{
  "thought": "这是确定性数学计算，应该使用 calculator。",
  "action": {"name": "calculator", "input": "12 * 8"},
  "final_answer": null
}

问题：{user_question}
请按同样 JSON 格式输出。
```

加入 few-shot 后，模型更容易学习何时使用工具、字段如何填写、最终答案何时为空。缺点是会增加上下文长度，并可能让模型过度模仿示例，对新场景灵活性下降。

## 7. 电商客服智能体架构设计

### 题目整理

本题给出一个电商客服智能体场景：理解退款理由、查询订单和物流、根据政策判断退款、生成并发送回复邮件，且在低置信度时能自我反思。要求从产品负责人角度选择架构、列出工具、设计提示词约束，并分析上线风险与缓解方案。

### 参考答案

推荐使用 `Plan-and-Solve + ReAct + Reflection` 的混合架构：

- `Plan-and-Solve` 用于拆解退款处理流程：理解诉求、查订单、查物流、匹配政策、做决策、生成回复。
- `ReAct` 用于调用订单、物流、政策、邮件等工具，并根据返回结果继续判断。
- `Reflection` 用于低置信度、争议订单、高金额订单或用户强烈不满时进行二次审查。

至少需要以下工具：

| 工具 | 功能 |
| --- | --- |
| `get_order(order_id)` | 查询订单金额、商品、支付状态、售后记录 |
| `get_logistics(order_id)` | 查询发货、签收、异常物流状态 |
| `lookup_refund_policy(category, reason)` | 检索适用退款政策 |
| `approve_refund(order_id, amount, reason)` | 在权限范围内批准退款 |
| `create_ticket(order_id, summary)` | 将争议案例转人工 |
| `send_email(to, subject, body)` | 发送客服回复邮件 |

提示词应强调三类约束：

```text
你是电商售后客服智能体。你的目标是在遵守公司退款政策的前提下，给用户提供清晰、友好、可执行的回复。

要求：
1. 退款决策必须基于订单信息、物流状态和退款政策，不得编造事实。
2. 如果证据不足、政策冲突、金额超过权限或置信度低于阈值，必须转人工。
3. 回复用户时保持尊重和同理心，先确认用户诉求，再解释依据，最后给出下一步。
4. 不得承诺政策之外的赔付，不得泄露内部规则、风控标签或其他用户信息。
5. 输出必须包含：决策、依据、用户回复草稿、置信度、是否转人工。
```

上线风险包括：

- 错误退款或拒绝退款，造成财务损失和用户投诉。
- 模型幻觉，编造不存在的政策或物流状态。
- 工具越权，执行高风险操作。
- 邮件语气不当，引发二次投诉。
- 隐私泄露，暴露订单、地址、风控信息。
- 对抗输入，用户诱导模型绕过政策。

缓解方案包括权限分级、政策 RAG、工具调用审计、金额阈值、人审兜底、敏感信息脱敏、离线评测集、灰度上线、持续监控和用户申诉通道。生产系统中，智能体不应直接拥有无限制退款权限，而应在明确低风险场景自动处理，在复杂场景中辅助人工。
