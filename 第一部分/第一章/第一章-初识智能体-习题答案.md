# 第一章 初识智能体：习题分析与参考答案

参考文章：[Datawhale Hello Agents 第一章 初识智能体](https://datawhalechina.github.io/hello-agents/#/./chapter1/%E7%AC%AC%E4%B8%80%E7%AB%A0%20%E5%88%9D%E8%AF%86%E6%99%BA%E8%83%BD%E4%BD%93)

说明：以下答案为参考答案。部分开放题没有唯一标准答案，重点在于能否结合智能体的定义、PEAS 框架、Workflow 与 Agent 的差异，以及系统 1 / 系统 2 思维方式进行合理分析。

## 1. 判断四个 Case 是否属于智能体

| Case | 是否属于智能体 | 类型分析 | 理由 |
| --- | --- | --- | --- |
| A 超级计算机 | 通常不属于 | 只是算力载体 | 单纯拥有高算力不等于智能体。它没有明确目标、传感器、执行器，也没有自主的“感知-决策-行动”闭环。 |
| B 特斯拉自动驾驶 | 属于 | 反应式 / 基于模型 / 混合式智能体 | 它通过摄像头、雷达等感知环境，在毫秒级做刹车或变道决策。紧急避障偏反应式，但也依赖道路、车辆、障碍物等内部世界模型。 |
| C AlphaGo | 属于 | 基于目标 / 基于效用 / 学习型 / 规划式智能体 | 目标是赢棋，会评估局面效用并规划未来多步。它也通过自我对弈和强化学习改进策略。 |
| D ChatGPT 智能客服 | 属于，前提是接入工具并能执行动作 | LLM 驱动 / 混合式 / 神经符号主义智能体 | 它需要理解投诉、查询订单、分析原因、调用工具、生成方案并安抚用户，符合现代 LLM Agent 的工具调用和动态决策特征。 |

### 小结

判断一个系统是不是智能体，不能只看它是否“聪明”或是否有强大的计算能力，而要看它是否具备：

- 明确目标；
- 对环境的感知能力；
- 根据感知进行决策的能力；
- 对环境产生影响的行动能力；
- 一定程度的自主性。

## 2. 智能健身教练的 PEAS 分析

| PEAS 要素 | 设计 |
| --- | --- |
| Performance 性能度量 | 训练目标达成度、运动安全性、动作标准度、用户坚持率、疲劳控制、受伤率降低、饮食建议有效性、用户满意度。 |
| Environment 环境 | 用户身体状态、运动场地、器械条件、天气、时间安排、历史训练记录、饮食摄入、睡眠和压力情况。 |
| Actuators 执行器 | 语音指导、手机 App 推送、训练计划调整、动作纠正提示、饮食建议、风险预警、休息提醒。 |
| Sensors 传感器 | 心率手环、运动手表、摄像头、加速度计、陀螺仪、用户输入、体重体脂数据、训练日志。 |

### 环境特性分析

智能健身教练所处环境具有以下特征：

- 部分可观察：智能体无法完全知道用户真实疲劳、肌肉状态、饮食执行情况和心理状态。
- 随机性：同样的训练计划对不同用户效果不同，用户状态也会受睡眠、压力、饮食等因素影响。
- 动态性：用户心率、动作姿态、疲劳程度会实时变化。
- 序贯性：今天的训练会影响明天的恢复情况和后续训练安排。
- 社会性 / 多主体性：用户本人、医生、真实教练、饮食平台等都可能影响最终训练决策。

## 3. Workflow 与 Agent 处理退款问题

### 方案 A：Workflow

Workflow 的优点是规则清晰、稳定可控、成本低、容易审计，适合高频、标准化、风险边界明确的退款场景。

它的缺点是比较僵硬，遇到复杂情况、特殊用户、灰色案例时容易误判。例如用户描述不规范、订单信息有异常、商品状态复杂时，固定规则可能无法给出合适处理。

### 方案 B：Agent

Agent 的优点是能理解上下文，可以综合用户历史、商品状况、聊天记录和退款政策解释，适合处理复杂投诉和非标准案例。

它的缺点是可解释性和一致性较弱，可能产生幻觉、越权判断或不合规回复，因此需要工具权限控制、规则约束和人工兜底。

### 推荐方案：Workflow + Agent 混合方案

更合理的做法是采用混合方案：

- 低风险、标准化申请走固定 Workflow 自动处理；
- 高金额、特殊商品、争议订单、疑似欺诈等进入 Agent 辅助分析；
- 关键退款决策由人工审核或规则网关确认；
- Agent 主要负责理解复杂语义、补全信息、解释政策和生成处理建议。

这样既保留 Workflow 的稳定性，又利用 Agent 处理复杂语义和边界案例。

## 4. 智能旅行助手扩展设计

### 增加记忆功能

可以为智能旅行助手新增用户画像存储，例如：

```text
memory = {
  "偏好": "历史文化",
  "预算": "500 元",
  "不喜欢": "人多景点"
}
```

在每轮对话中，系统可以检索与当前任务相关的记忆，并把它们加入 Prompt。工具调用后也可以更新记忆。

可以设计两个工具：

- `save_memory(key, value)`：保存用户偏好或历史信息；
- `retrieve_memory(query)`：根据当前任务检索相关记忆。

### 门票售罄时推荐备选方案

新增工具：

```text
check_ticket(attraction, date)
```

推荐景点前先查询门票。如果返回售罄，则调用：

```text
get_alternative_attraction(city, preference, weather)
```

然后根据用户偏好、预算、天气、交通距离等因素给出替代方案。

### 连续拒绝 3 次后的反思机制

可以维护两个状态：

```text
rejection_count
rejection_reasons
```

当用户连续拒绝 3 次后，触发反思步骤，让智能体总结失败原因，例如：

- 预算不符；
- 距离太远；
- 用户不喜欢热门景点；
- 推荐类型与用户偏好不匹配；
- 行程节奏太紧。

整体循环可以设计为：

```text
Thought -> Action -> Observation -> Memory Update -> Reflection -> Next Action
```

这样智能体就能从失败反馈中调整推荐策略，而不是机械地继续给出类似建议。

## 5. 系统 1 / 系统 2 的应用场景

以“医疗初诊助手”为例。

### 系统 1：快速直觉式判断

系统 1 适合处理快速模式识别任务，例如：

- 识别症状关键词；
- 判断是否存在紧急风险；
- 匹配常见疾病模式；
- 检测明显风险信号，如胸痛、呼吸困难、高烧不退等。

系统 1 的优势是速度快，适合初步筛查和风险分流。

### 系统 2：慢速推理式判断

系统 2 适合处理需要多步推理的任务，例如：

- 结合病史、年龄、用药和检查结果进行分析；
- 进行鉴别诊断；
- 解释可能原因；
- 生成就医建议；
- 判断是否需要转人工医生。

系统 2 的优势是更审慎、更结构化，适合复杂场景。

### 协同方式

系统 1 可以先快速筛查和分流。如果发现高风险，立即建议急诊或人工介入；如果是普通问题，再交给系统 2 做结构化问诊、推理和建议生成。

在医疗场景中，最终输出还应经过医学规则、安全策略和人工审核机制约束，避免错误建议造成风险。

## 6. 智能体局限与评估

### 为什么智能体会产生幻觉？

智能体产生幻觉，主要有以下原因：

- LLM 本质上是在生成高概率文本，不是天然的事实数据库；
- 模型训练数据可能过时、不完整或存在错误；
- 工具返回结果可能缺失、过期或解析错误；
- 上下文过长时，模型可能遗漏或误用信息；
- 如果缺少检索、验证和约束，模型容易生成看似合理但实际错误的答案。

因此，实际系统中通常需要结合 RAG、工具调用、事实校验、引用来源、权限控制和人工审核来降低幻觉风险。

### 为什么不能没有最大循环次数？

如果没有最大循环次数，智能体可能陷入无限循环，例如：

- 反复调用同一个工具；
- 不断修正计划却不结束；
- 在错误观察结果上持续推理；
- 因为格式解析失败而重复尝试；
- 无法判断任务已经完成。

因此，Agent 系统需要设置最大步数、超时机制、停止条件、异常处理和人工接管策略。

### 评估智能体为什么不能只看准确率？

准确率只能衡量最终答案是否正确，不能完整反映智能体的整体能力。评估智能体还应关注：

- 任务完成率；
- 工具调用正确率；
- 规划质量；
- 鲁棒性；
- 安全性；
- 成本；
- 响应时延；
- 可解释性；
- 用户满意度；
- 是否能从反馈中修正行为。

因此，对智能体的评估应该同时关注结果、过程、安全和用户体验，而不是只看单一准确率指标。
