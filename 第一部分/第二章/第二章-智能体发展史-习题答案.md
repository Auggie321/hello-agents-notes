## 1. 物理符号系统假说与符号主义智能体

### 题目要点

分析物理符号系统假说中的“充分性论断”和“必要性论断”；结合符号主义智能体的实践问题，说明这些问题如何挑战该假说；进一步判断大语言模型驱动的智能体是否符合该假说。

### 参考答案

物理符号系统假说认为，智能可以通过符号及其操作过程来实现。

- 充分性论断：只要一个系统具备操作物理符号的能力，它就原则上具备产生通用智能行为的充分条件。
- 必要性论断：任何表现出通用智能行为的系统，本质上都应当是某种物理符号系统。

符号主义智能体在专家系统、SHRDLU 等早期系统中取得过重要成果，但这些成果大多依赖封闭环境、人工规则和清晰边界。一旦进入真实世界，就会暴露出明显局限：

- 知识获取瓶颈：系统知识需要专家手工编码，成本高、更新慢、难以规模化。
- 常识问题：真实世界依赖大量默认常识，而符号系统通常只知道被明确写入的规则。
- 框架问题：在动态世界中，系统很难高效判断一个行动后哪些状态改变、哪些状态不变。
- 系统脆弱性：遇到规则之外的输入或异常情况时，系统容易失效。

这些问题并不一定彻底否定“符号操作可以产生智能”，但它们挑战了“仅靠显式符号系统就足以实现通用智能”的强版本观点。

LLM 驱动的智能体与物理符号系统假说存在复杂关系。表面上，LLM 处理的是文本符号，并可以进行代码、逻辑、工具调用和规划；但其核心能力来自大规模参数中的分布式表示，而不是人工设计的显式符号规则。因此，LLM Agent 更像是神经表示与符号操作的结合体：它部分符合“能操作符号”的描述，但不能简单归入传统符号主义智能体。

## 2. MYCIN 与医疗专家系统的现实局限

### 题目要点

分析 MYCIN 这类医疗专家系统为何没有大规模进入临床；设想现代医疗诊断智能体应如何克服它的局限；举例说明哪些场景下规则专家系统仍然优于深度学习。

### 参考答案

MYCIN 在特定医疗诊断任务上表现突出，但没有被大规模临床采用，原因不只在技术层面。

技术方面：

- 系统依赖手工知识库，医学知识更新成本高。
- 推理范围窄，难以覆盖复杂合并症、罕见病和非典型病例。
- 无法自然整合影像、病历、检验报告、用药史等多模态数据。
- 对输入质量敏感，错误或遗漏信息会导致诊断偏差。

伦理与法律方面：

- 如果系统建议错误，责任归属不清。
- 医疗决策关系生命安全，不能只依赖黑箱或半自动系统。
- 患者隐私、数据授权和审计合规要求很高。

用户接受度方面：

- 医生可能不信任系统，担心影响专业判断。
- 患者也可能不愿接受由机器直接给出的诊断建议。
- 医疗流程复杂，系统若不能嵌入真实工作流，就难以持续使用。

如果设计现代医疗诊断智能体，应采用“辅助决策”而不是“替代医生”的定位。可行架构包括：

- 数据层：接入结构化病历、检验指标、医学影像、用药记录和医学指南。
- 检索层：使用 RAG 检索最新指南、药品说明书和权威医学知识库。
- 推理层：用 LLM 做病情摘要、鉴别诊断、风险分层和解释生成。
- 规则层：对禁忌症、危急值、药物相互作用等高风险规则进行硬约束。
- 审核层：关键建议必须由医生确认，并保留完整日志。
- 安全层：设置不确定性提示、转人工机制、隐私保护和权限控制。

规则专家系统至今仍适合规则稳定、可解释性强、容错空间小的垂直场景。例如：

- 税务与财务合规检查。
- 航空、铁路等安全规程校验。
- 药物相互作用和禁忌症提醒。
- 工业设备故障规则诊断。
- 银行业反洗钱初筛规则。

这些场景中，规则清晰、可审计、可追责，专家系统往往比纯深度学习更可靠。

## 3. 扩展 ELIZA 并分析规则对话系统局限

### 题目要点

在简化版 ELIZA 基础上扩展更多规则；加入简单上下文记忆；比较扩展后的 ELIZA 与 ChatGPT 的本质差异；解释规则方法在开放域对话中为何会出现组合爆炸。

### 参考答案

可以为 ELIZA 添加工作、学习、爱好等场景规则，例如：

```python
rules = [
    (r".*我叫(.+)", "很高兴认识你，{name}。"),
    (r".*我在(.+)工作.*", "你在{job}工作时，最常遇到的压力是什么？"),
    (r".*我正在学习(.+)", "学习{subject}时，你觉得最困难的部分是什么？"),
    (r".*我喜欢(.+)", "你喜欢{hobby}多久了？它给你带来了什么感受？"),
    (r".*我担心(.+)", "你为什么会担心{worry}？")
]
```

简单上下文记忆可以用字典保存用户信息：

```python
memory = {}

def update_memory(user_input):
    if "我叫" in user_input:
        memory["name"] = user_input.split("我叫", 1)[1].strip()
    if "我在" in user_input and "工作" in user_input:
        memory["job"] = user_input.split("我在", 1)[1].split("工作", 1)[0].strip()

def recall_memory():
    if "name" in memory:
        return f"刚才你提到你叫{memory['name']}。"
    return "你可以继续说说。"
```

扩展后的 ELIZA 与 ChatGPT 至少有以下本质差异：

| 维度 | 扩展 ELIZA | ChatGPT |
| --- | --- | --- |
| 理解方式 | 关键词匹配和模板替换 | 基于大规模语料学习到的上下文表示 |
| 泛化能力 | 只能覆盖预设规则 | 可处理大量未见过的表达方式 |
| 记忆能力 | 只能保存少量显式字段 | 可在上下文中综合多轮信息 |
| 推理能力 | 基本没有真正推理 | 可进行多步分析、归纳和规划 |
| 可维护性 | 规则越多越难维护 | 主要通过模型、提示、工具和检索扩展 |

规则方法在开放域对话中会遇到组合爆炸。假设一个系统要覆盖 `n` 类主题，每类主题有 `m` 种表达方式，还要考虑 `k` 种上下文状态，那么规则数量可能接近 `n * m * k`。如果再考虑多轮对话，状态组合会进一步指数级增长。真实语言还有同义表达、指代、省略、歧义和情绪变化，靠人工规则几乎无法穷尽。

因此，ELIZA 能展示早期对话系统的巧妙设计，但它并不真正理解语言。它的价值在于帮助我们看清规则驱动方法的边界。

## 4. “心智社会”与多智能体系统

### 题目要点

讨论“搭建积木塔”案例中某个子智能体失效后的影响；分析去中心化架构的优劣；比较“心智社会”理论与现代多智能体系统；判断该理论在 LLM 时代是否仍有价值。

### 参考答案

如果 `GRASP` 这类负责抓取动作的子智能体失效，整个积木塔系统会受到明显影响。即使规划模块知道下一步要移动某块积木，执行层也无法完成抓取动作，任务会停在行动环节。如果系统有冗余机制，可能调用备用抓取策略；如果没有，就会整体失败。

去中心化架构的优势：

- 模块职责清晰，每个子智能体只需处理局部问题。
- 系统更容易扩展，可以增加新的能力模块。
- 局部模块可以独立优化和替换。
- 多个子智能体协作时，可能涌现出复杂行为。

去中心化架构的劣势：

- 协调成本高，需要解决任务分配、通信和冲突问题。
- 局部失败可能影响整体任务。
- 全局最优难以保证。
- 调试难度较高，因为错误可能来自模块本身，也可能来自模块交互。

“心智社会”与现代多智能体系统有明显关联。CAMEL、MetaGPT、CrewAI 等系统也会把复杂任务拆成多个角色或智能体，让它们通过协作完成任务。例如，一个软件项目可以拆成产品经理、架构师、工程师、测试等角色。

二者的区别在于：

- 心智社会更偏认知理论，关注智能如何由简单过程组合产生。
- 现代多智能体框架更偏工程系统，关注任务编排、角色分工、工具调用和结果交付。
- 明斯基设想的许多子智能体可以非常简单，而现代 LLM Agent 往往本身就具备强推理和语言能力。

LLM 时代并没有让“心智社会”过时。相反，它让这个理论有了新的工程实现方式。即使单个 LLM 很强，复杂任务仍然需要分工、专门化、协作、批判和反思。现代 Agent 系统中的 planner、executor、critic、memory、tool user 等模块，都可以视为“心智社会”思想的延续。

## 5. 强化学习、监督学习与序贯决策

### 题目要点

用 AlphaGo 说明强化学习的试错机制；解释强化学习为何适合序贯决策；比较训练超级马里奥智能体时监督学习和强化学习的数据需求；说明强化学习在大语言模型训练中的作用。

### 参考答案

强化学习的核心是智能体通过与环境交互获得奖励，并逐步学习更优策略。以 AlphaGo 为例，它通过棋局状态观察当前局面，选择落子动作，经过对弈获得胜负结果或局面价值反馈，然后调整策略，使未来更可能选择高胜率行动。

强化学习适合序贯决策，因为很多任务的结果不是由单一步骤决定的，而是由一系列行动共同导致的。例如围棋中一步棋的价值可能要几十手后才显现。强化学习能够处理延迟奖励、长期收益和探索-利用权衡。

监督学习和强化学习的数据需求不同：

- 监督学习需要大量“输入-正确答案”样本。例如给定棋盘状态，标注人类高手下一步怎么走。
- 强化学习不一定需要标准答案，它需要环境反馈。例如智能体自己下棋、试错，根据胜负和奖励学习策略。

训练超级马里奥智能体时：

- 如果使用监督学习，需要收集大量人类玩家的游戏画面、按键操作和状态标签，让模型模仿人类行为。
- 如果使用强化学习，需要一个可交互游戏环境，并设计奖励，例如前进距离、得分、存活时间、通关奖励、死亡惩罚。

对超级马里奥这类游戏，强化学习通常更合适，因为它天然是序贯决策任务，智能体需要通过探索学习跳跃、避障、攻击和路线选择。

在大语言模型训练中，强化学习常用于让模型输出更符合人类偏好和安全要求。例如 RLHF 会先收集人类偏好数据，训练奖励模型，再通过强化学习优化语言模型，使其回答更有帮助、更安全、更符合指令。近年的 RLVR 等方法也会在数学、代码等可验证任务中利用明确奖励改进推理能力。

## 6. 预训练-微调范式与知识获取方式变化

### 题目要点

分析预训练为何缓解了符号主义时代的知识获取瓶颈；比较符号知识和预训练知识的表示方式；讨论互联网数据带来的问题及缓解办法；判断预训练-微调范式是否可能被取代。

### 参考答案

符号主义时代的知识获取依赖专家手工编码，知识进入系统的方式是显式规则、逻辑表达式或知识库条目。这个过程昂贵、缓慢，且很难覆盖开放世界。

预训练改变了知识获取方式。模型通过海量文本、代码、图像等数据自监督学习，从数据分布中自动吸收语言规律、事实关联、常识模式和任务结构。它不需要每条知识都由专家写成规则，因此显著缓解了知识获取瓶颈。

二者的知识表示方式有本质区别：

| 维度 | 符号主义 | 预训练模型 |
| --- | --- | --- |
| 表示形式 | 显式规则、符号、逻辑关系 | 分布式向量和模型参数 |
| 获取方式 | 人工编码 | 从大规模数据中学习 |
| 可解释性 | 较强 | 较弱 |
| 泛化方式 | 基于规则匹配和推理 | 基于表示学习和统计泛化 |
| 更新方式 | 修改知识库或规则 | 微调、继续训练、检索增强、外部工具 |

互联网数据也会带来问题：

- 偏见和歧视：数据中存在社会偏见，模型可能学习并放大。
- 错误和谣言：网络内容质量不一，事实可能不可靠。
- 数据过时：模型参数中的知识会滞后。
- 版权和隐私风险：训练数据可能包含受保护或敏感内容。
- 长尾知识不足：小众领域或低资源语言覆盖不足。

缓解方法包括：

- 数据清洗、去重、质量过滤和隐私脱敏。
- 安全对齐、偏见评估和红队测试。
- 使用 RAG 接入权威、实时、可追溯知识源。
- 引入工具调用，让模型查询数据库、计算器和业务系统。
- 在高风险场景加入规则约束和人工审核。

预训练-微调范式可能不会很快消失，但会不断演化。未来可能更强调检索增强、工具使用、持续学习、多模态世界模型、智能体自我改进和可验证奖励训练。也就是说，它未必被完全取代，更可能成为更大系统中的基础能力层。

## 7. 智能代码审查助手的三代方案比较

### 题目要点

设想在符号主义时代、无大语言模型的深度学习时代、当前 LLM Agent 时代分别如何设计“智能代码审查助手”；比较三种方案，说明智能体技术演进如何让该任务从困难变得可行。

### 参考答案

如果在 1980 年代的符号主义时代设计代码审查助手，主要会采用规则系统和静态分析：

- 编写语法规则、代码风格规则和缺陷模式规则。
- 使用编译器、控制流图、数据流分析检查潜在问题。
- 通过专家规则识别空指针、未初始化变量、资源泄漏等问题。

主要困难是规则覆盖有限，无法真正理解业务意图、代码上下文和开发者设计取舍。系统可以检查形式化问题，但很难概括实现逻辑，也难以发现复杂语义 Bug。

如果在 2015 年左右的深度学习时代设计，可以结合传统静态分析和神经网络模型：

- 用代码 token、AST、图神经网络等方式表示代码。
- 训练模型识别漏洞、重复代码、坏味道或命名问题。
- 使用历史提交和人工 review 数据进行监督学习。

这种方案比纯规则更能学习模式，但仍有明显限制：模型依赖标注数据，跨项目泛化弱，对长上下文和业务需求理解不足，也难以给出高质量自然语言解释。

在当前 LLM Agent 时代，可以设计为模块化智能体架构：

- 感知模块：读取 PR diff、相关文件、测试结果、Issue、需求文档和代码规范。
- 记忆模块：保存项目架构、历史 review 偏好、团队规范和已知风险点。
- 规划模块：把审查任务拆成实现逻辑概括、风险检查、测试覆盖检查、风格检查和改进建议。
- LLM 推理模块：理解代码意图，跨文件分析变更影响，生成审查意见。
- 工具模块：调用静态分析器、类型检查器、测试命令、依赖漏洞扫描、代码搜索和 Git 历史。
- 执行模块：运行工具、收集结果、过滤误报。
- 反思模块：复核评论是否具体、可执行、不过度阻塞。
- 输出模块：按严重程度给出文件位置、问题描述、影响和修改建议。

三代方案对比如下：

| 时代 | 核心能力 | 优势 | 局限 |
| --- | --- | --- | --- |
| 符号主义 | 规则和静态分析 | 可解释、稳定、适合明确规则 | 难理解语义和业务意图 |
| 深度学习 | 从代码数据中学习模式 | 能发现部分统计规律 | 依赖标注，解释弱，长上下文弱 |
| LLM Agent | 语言理解、工具调用、规划和记忆 | 能综合上下文并生成可读建议 | 仍需防幻觉、控权限、做验证 |

技术演进的关键在于：早期系统只能检查“规则上错了什么”，深度学习开始学习“历史上哪些代码像问题”，而 LLM Agent 可以进一步理解“这次变更想做什么、可能影响哪里、应该如何验证”。因此，智能代码审查从“几乎只能做局部静态检查”，逐渐变成“可以辅助完整 review 流程”的可行任务。
